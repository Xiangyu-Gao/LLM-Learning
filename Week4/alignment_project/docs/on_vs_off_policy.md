# On-Policy vs. Off-Policy in RL for LLMs

## The Core Distinction

In reinforcement learning, **on-policy** and **off-policy** describe the relationship between the policy used to *collect data* and the policy being *updated*.

| Term | Definition |
|------|-----------|
| **On-policy** | The policy being trained IS the policy that generated the training data |
| **Off-policy** | The policy being trained is DIFFERENT from the policy that generated the data |

---

## Why This Matters for Language Models

LLMs generate entire sequences autoregressively.  Each token generation is a "action" in RL terms.  The distribution of sequences depends critically on which policy generated them.

If you train on sequences from policy A but update policy B, the importance weights (π_B / π_A) can be large, leading to high variance or instability.

---

## Classification of Alignment Methods

### DPO — Off-Policy
```
Data collection:  some other model / human annotators
Training:        gradient updates on the fixed dataset
Policy relationship: DIFFERENT — the preference dataset was collected
                     from a policy that is NOT the model being trained
```

DPO is **offline** and **off-policy**.  The dataset `data/preferences/train.jsonl`
was generated by heuristics / gold labels — it is static.  DPO trains a new policy
on this fixed distribution.

**Implication:** If the policy is far from the data distribution, log-probs are
poorly calibrated and the implicit reward model is unreliable.

---

### PPO — On-Policy
```
Data collection:  current policy π_θ generates completions
Training:        gradient update using those fresh completions
Policy relationship: SAME — the policy that collected the data IS being updated
```

PPO is **online** and **on-policy** (within an epoch).  Before each update,
fresh rollouts are generated from the current policy.

**Nuance:** PPO uses multiple gradient updates per rollout batch (`ppo_epochs`).
After the first update, the policy has changed; subsequent updates in the same
epoch are technically using "stale" data.  The clipped importance ratio handles
this safely (the clip prevents large updates when ratio ≠ 1).

---

### GRPO — On-Policy
```
Data collection:  current policy π_θ generates G completions per prompt
Training:        group-relative advantage + clipped policy gradient
Policy relationship: SAME — on-policy generation at start of every step
```

GRPO is **on-policy** in the same sense as PPO.

**What "old policy" means in GRPO:**
The importance ratio uses `π_old` — but `π_old` is a **snapshot of π_θ taken at the
beginning of the current optimisation step**, not a policy from a previous epoch.
This is standard in all PPO-style algorithms.

---

## When People Say "Off-Policy" for DPO

You will hear practitioners say:

> "DPO is off-policy because we can't guarantee the preference data came from the current policy."

This is correct.  The more precise term is **"offline RL"** — the data is a fixed
dataset, not collected by the current policy.  In contrast, PPO and GRPO are
**"online RL"** — they continually generate fresh data.

---

## Interview Q&A

### Q: Is GRPO on-policy?

**A:** Yes, strictly speaking.  GRPO generates G fresh completions from the CURRENT
policy parameters at the start of every training step.  These rollouts are used
immediately for the gradient update.  After the update, the policy parameters have
changed and entirely new rollouts are generated for the next step.

The "old policy" (π_old) in the clipped importance ratio is merely a snapshot of
π_θ taken at the START of the same step — it is not a policy from a previous epoch
or a separate deployment.

Nuance: if `num_ppo_epochs > 1` (multiple gradient steps per rollout batch), the
later updates are "near-on-policy" rather than strictly on-policy.  The PPO clip
(1−ε, 1+ε) exists precisely to keep these near-on-policy updates safe.

Compare with DPO: DPO is **off-policy** because it trains on a FIXED dataset of
(prompt, chosen, rejected) tuples collected from an earlier model or human
annotators — never from the policy currently being trained.

---

### Q: When would you prefer on-policy over off-policy?

**A:**
- **On-policy (PPO/GRPO):** When you need the policy to explore and the reward
  function is computable at inference time (e.g., code execution, math verification,
  format checking).  Online feedback continuously corrects the policy's distribution.
- **Off-policy (DPO):** When you have a high-quality fixed preference dataset and
  cannot afford online generation (which requires keeping the policy in memory for
  inference).  DPO is much simpler to implement and debug, making it the default
  choice for many production alignment runs.
